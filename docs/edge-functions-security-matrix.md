# Edge Functions Security Matrix

This document is the S1 inventory of Supabase Edge Functions.

Classifications:

- public: can be called without a logged-in user, but must remain read-only or low-risk.
- authenticated: requires a valid user JWT.
- admin: requires admin role verification.
- internal: called by backend automation or trusted service.
- webhook: called by an external provider and must verify a signature or shared secret.
- scheduled: called by a scheduler.
- service-role: uses the service role key server-side.

## Global findings

- Total functions: 27
- Service role is used by several backend-only functions.
- The `mcp` function is auto-generated by Lovable and should not be manually edited.
- No function has an explicit timeout pattern detected. This should become a future standard.
- P0-sensitive functions must stay under separate finance/security review.

## Functions requiring explicit follow-up before public exposure

1. `mcp`: auto-generated and read-only by design, but no explicit auth/CORS/logging was detected in the generated file.
2. `initiate-mobile-money-payment`: payment surface; any change requires finance/P0 review.
3. `mobile-money-webhook`: must keep signature, anti-replay and provider reference checks mandatory.
4. `api-gateway`: should remain registry-driven with explicit permissions per endpoint.
5. `track-referral-click`: public tracking should have rate limiting and abuse controls.

## Standard to adopt for future hardening

Every Edge Function should document:

- allowed method
- caller type
- JWT requirement
- role requirement
- secret/signature requirement
- maximum payload size
- idempotency key
- retry behavior
- timeout budget
- log fields
- sensitive fields redacted
- metrics emitted
- alert condition
- P0 exposure: none/read-only/finance-critical
